## 1. 프로젝트 개요

프로젝트 제목은 "비정형 얼굴 이미지 기반 감정 인식을 활용한 결과 분기형 인터랙션 시스템". 4주 안에 얼굴 표정 분류 모델 한 개를 학습시키는 과제로 끝내지 않고, 분류 결과가 인터랙션의 분기점으로 이어지는 서비스까지 묶어내는 쪽에 무게를 실었다. 리더보드 싸움으로 끝나는 팀이 많아 보였고, 같은 모델도 어떤 상호작용에 꽂히느냐에 따라 가치가 달라진다는 판단에서다.

핵심은 방향을 뒤집은 데 있다. 기존 FER이 "표정 입력 → 감정 분석"이라면, 우리 시스템은 "상황(스토리) 제시 → 사용자가 해당 감정의 표정을 만들어냄 → 모델이 분류 → 결과에 따라 스토리 분기"로 흐른다. 1차 타깃은 자폐 스펙트럼 아동의 감정 학습 도구, 응용 범위는 표정만으로 진행되는 스토리형 게임까지 넓게 잡았다. 분류 대상은 anger·happy·panic·sadness 4클래스, 데이터는 이스트소프트 제공 얼굴 이미지 7,196장, 팀은 2조 5인 (김대원·박재성·반주형·하창수·엄정원) 구성이다.

## 2. 문제 정의와 서비스 가설

### 2.1 문제 정의 — 감정 카드의 단방향성

자폐 스펙트럼 아동은 상황과 감정을 연결하는 데 정형 발달 아동보다 긴 학습 시간을 필요로 한다. 치료·교육 현장에서 널리 쓰이는 감정 카드 학습은 카드 위의 표정을 보여주고 이름을 외우게 하는 방식으로 정리된다. 입력이 표정, 출력이 감정 라벨. 이 방향은 "본 표정을 구분"까지는 길러내지만, 실제 사회적 상호작용에서 필요한 능력 — **상황을 읽고 그에 맞는 표정을 만들어내는** 역방향 훈련 — 에는 거의 개입하지 못한다.

시장에 풀려 있는 FER 기반 서비스 대부분도 이 단방향 프레임 안에서 움직인다. 카메라로 표정을 받고, 감정 태그를 출력하고, 끝. 사용자가 "자연스럽게 감정을 표현할 수 있다"는 전제가 깔려 있어서, 표정 생성 자체가 어려운 아동 사용자층은 시스템 바깥에 놓인다.

### 2.2 서비스 가설 — 세 개의 H

가설은 세 가지로 쪼갤 수 있다.

**H1 (매칭 가설)**. 챗봇이 특정 감정이 자연스럽게 드러나는 상황 (예: "친구가 네 생일에 깜짝 선물을 줬어")을 스토리 형태로 제시하면, 사용자는 그 상황에 맞는 표정 (기쁨)을 만들어낸다. 감정 카드의 단방향 루프가 역방향으로 작동하는 구조다.

**H2 (분기 가설)**. 모델이 출력하는 (예측 클래스, confidence) 쌍을 이용해 스토리를 분기시킨다. `pred == expected`이고 confidence가 충분히 높으면 다음 장면으로, 어긋나면 힌트 장면이나 재시도 루프로. 게임형 애플리케이션에서는 클릭·음성 대신 표정 자체가 입력 모달이 된다.

**H3 (자가발전 가설)**. 챗봇이 제시한 상황에는 **정답 감정 라벨이 사전에 설계되어 있다**. 사용자가 그 상황에서 실제로 해당 감정 표정을 지었다면 — `conf ≥ 0.85` AND `pred == expected` — 그 (이미지, 기대 라벨) 쌍은 라벨링 비용 0의 신규 학습 데이터가 된다. 이 샘플을 rehearsal buffer에 쌓아 두고, 야간 배치에서 부모·치료사의 검수를 거쳐 승인된 샘플만 점진 재학습에 투입한다. 프로젝트 기간 안에 단계 1 (파이프라인 설계 + 조건부 수집 코드)까지 완결하고, 단계 2 (실제 재학습 루프)는 향후 과제로 남긴다.

### 2.3 기존 FER과의 차별점

| 축 | 기존 FER | 본 프로젝트 |
|---|---|---|
| 방향 | 표정 → 감정 | **상황 → 표정 유도 → 감정** |
| 전제 사용자 | 이미 표정을 낼 수 있는 사람 | 표정 생성 자체가 학습 목표인 아동 |
| 라벨 공급 | 사전에 주어진 라벨셋 | **스토리의 기대 감정이 weak label** |
| 데이터 선순환 | 없음 | 조건부 수집 + HITL 승인 → 자가발전 |

### 2.4 연구 질문과 왜 이 데이터·왜 4-class

연구 질문은 두 개로 좁혔다. 첫째, 상황 → 표정 유도 → 감정 분류 파이프라인이 데모 수준에서 쓸 만한 정확도로 동작하는가. 둘째, 서비스 런타임에 쌓이는 "상황-표정 정답쌍"이 모델을 실제로 개선시킬 여지가 있는가.

데이터셋 선택 이유는 두 가지다. 한국인 얼굴 비율이 높아 1차 타깃 사용자 (국내 아동)의 분포와 가깝다는 점, 그리고 4-class (anger·happy·panic·sadness)가 자폐 아동 감정 학습 현장에서 가장 기초로 다뤄지는 1차 감정군과 겹친다는 점. 혼합감정·강도까지 확장하면 학습 난도가 급격히 올라가므로, MVP는 4-class 고정으로 간다.

## 3. 데이터셋

### 3.1 출처와 규모

AI-Hub / 이스트소프트에서 제공한 한국인 얼굴 감정 이미지 데이터셋을 사용한다. 전체 7,196장이며 train 5,996 / val 1,200으로 이미 분할되어 있다. 클래스별로는 anger 1,800, happy 1,795, panic 1,801, sadness 1,800장 수준으로 거의 균등하다. 클래스 불균형에 따른 class_weight 조정은 현 단계에서는 불필요하다고 판단했다.

라벨은 JSON (euc-kr 인코딩) 형식으로 제공되며, 이미지 좌표 기준의 얼굴 bounding box (annot_A·B·C 3인 라벨러), 연령, 성별, 일반인/전문인 여부 같은 메타 정보를 담고 있다. 세그멘테이션은 별도 NPZ 파일로 제공되고 값 체계는 0=bg / 1=hair / 2=body / 3=face / 4=cloth / 5=etc. train 연령은 mean 26.4, median 20 (10~60 범위), 성별은 여 3,236 / 남 2,758, isProf는 일반인 3,187 / 전문인 2,807으로 치우침이 크지는 않다.

### 3.2 해상도

원본 이미지는 대체로 고해상도다. 너비는 최소 1,330 / 중앙값 3,088 / 최대 9,248 픽셀, 높이는 최소 1,000 / 중앙값 2,208 / 최대 6,936 픽셀. 종횡비는 중앙값 1.33에 0.75~2.22 분포. ResNet/EfficientNet 입력인 224×224로 바로 다운샘플링하면 얼굴 영역 외 정보가 과도하게 압축되므로, annot_A의 bbox로 얼굴을 먼저 crop한 뒤 resize하는 전처리가 점수와 직결된다고 봤다.

### 3.3 초기 시각화에서 발견한 문제

EDA 첫 라운드에서 이미지의 24% 가량이 화면에 180도 (혹은 90/270도) 뒤집혀 표시됐다. 파일 픽셀 배열 자체는 제조사 기본 방향이었고, 뒤집힘 정보는 EXIF Orientation 태그에만 들어 있었기 때문이다. PIL로 열면 뷰어와 다른 방향이 나오는 전형적인 케이스. 이게 남아 있으면 라벨의 bbox 좌표계가 이미지 픽셀과 어긋나고, TF 파이프라인 (`tf.io.decode_image`)는 EXIF를 따지지 않으므로 학습 단계에서 어긋남이 그대로 손실로 들어간다.

대응은 파이프라인 앞단에서 한 번에 정리하는 쪽을 택했다. `data_rot/` 디렉터리를 새로 만들고 이미지에만 `ImageOps.exif_transpose`를 적용해 픽셀을 실제 방향대로 회전, EXIF 태그는 제거했다. 라벨 JSON과 segmentation NPZ는 원래부터 EXIF 정규화 좌표계 (뷰어 기준)로 찍혀 있었기 때문에 추가 변환 없이 그대로 복사했다. 이 "라벨이 이미 정규화 좌표계"라는 가설이 맞는지 검증하는 작업이 남았고, 이걸 수치로 풀지 않으면 학습을 시작할 수 없었다.

### 3.4 MTCNN 기반 정합성 검증

검증은 다음 순서로 돌렸다. `data_rot/` 이미지에 대해 MTCNN으로 얼굴을 재검출하고, 그 bbox와 라벨 annot_A bbox의 IoU를 계산. 240장 샘플 기준 IoU 평균 0.959, IoU > 0.3 조건을 모두(240/240) 통과했다. 즉 라벨 bbox는 "뷰어에서 본 이미지" 기준으로 찍혀 있었고, 이미지만 EXIF 정규화하면 학습 시 좌표계가 맞아떨어진다는 의미다. 모양(shape) 일치는 7,196/7,196. 이 결과로 `rotate_bbox` / `rotate_mask_np` 같은 라벨 회전 로직을 굳이 추가하지 않기로 결정했다.

### 3.5 남은 이슈와 처리 방침

전체가 깔끔하지는 않다. bbox 좌표가 음수로 찍힌 케이스 9건 (train/anger 2, train/happy 1, train/panic 3, train/sadness 3)이 있었고, 그중 area ≤ 0으로 아예 유효하지 않은 bbox가 2건 섞여 있다. 이미지만 있고 라벨 JSON이 누락된 케이스도 2건 (train/happy 1, train/panic 1). 학습 단계에서 음수 bbox는 `[0, W] × [0, H]` 클립, 면적 0 이하 2건은 drop, 라벨 누락 2건은 skip으로 처리한다. 전체 맥락에서 0.2% 미만이라 전략 자체는 단순한 가드 수준에 그친다.

세그멘테이션 마스크의 shape은 이미지마다 달라서 (예: 3024×4032, 2320×3088, 1080×1440 등) 학습 파이프라인에서 이미지와 같이 224×224로 리사이즈하는 과정이 필요하다. seg 라벨 값도 일부 클래스는 0~4, 일부는 0~5로 달라서, 5 (etc)를 0 (bg)과 합칠지 별도 클래스로 남길지는 E5 (seg 활용 실험) 설계 시 결정한다.

### 3.7 라벨러 3인 불일치 — 성능 천장의 근거

데이터셋의 라벨은 annot_A·B·C 세 명의 라터가 각각 붙인 값을 갖는다. 학습 시 보통 annot_A 하나만 쓰지만, 세 값이 얼마나 일치하는지를 따로 뽑아 봤다. 전원 일치 (A = B = C) 비율 **0.6108**, 다수결 일치 (2/3 이상 동의) 비율 **0.9045**. 사람 라터조차 완벽히 동의하지 못하는 샘플이 약 40%에 이른다.

혼동 쌍 상위를 보면 어디가 어렵게 섞이는지 윤곽이 잡힌다. 슬픔 ↔ 상처 계열에서 729회로 가장 많았고, panic ↔ 놀람, anger ↔ 불쾌처럼 인접 감정 간 혼동이 상위권을 채웠다. 4-class로 정리할 때 이 인접 감정들이 panic·sadness 쪽에 뭉쳐 들어가므로, 모델 입장에서 학습 target 자체가 내재적 노이즈를 갖는다.

결론적으로 val_acc 천장은 **0.85~0.90 구간**으로 추정된다. 이미 5장에 정리한 대로 E5 (ViT-B/16, 0.8458)가 이 구간 하단에 도달했다. 이 지점을 돌파하려면 hard label 1개만 쓰는 표준 cross-entropy 대신, 3인 라벨 분포를 soft label로 삼아 KL divergence loss로 학습하는 방향이 유망하다. 현 보고서에서는 가능성 지적 수준으로 두고, 5/1 이후 추가 실험 후보로 등록했다. 상세 수치는 `results/annot_consistency.md`.

## 4. 모델링

### 4.1 트랙 구성 — 왜 3개를 병렬로 돌렸나

단일 아키텍처에 몰빵하지 않고 CNN / ViT / SigLIP 세 트랙을 나란히 굴렸다. 이유는 두 가지였다. 하나는 앙상블 다양성 확보 — 서로 다른 inductive bias를 갖는 모델을 섞어야 soft voting이 실질적으로 이득이 된다는 점. 또 하나는 아키텍처 커버리지. ViT-B/16은 Transformer 계열을 대표로 넣었고, SigLIP 트랙은 팀 자체 설계한 vision_encoder wrapper (yua-encoder)를 살리는 쪽으로 남겼다. CNN은 "데이터 7천 장" 규모에서 가장 안정적인 하한선을 깔아 주는 역할.

### 4.2 CNN 트랙 (ResNet50, EfficientNet-B0)

ImageNet pretrained weight에서 출발, 초기에는 FC layer만 학습하는 frozen 구조로 놓고 (E1) → `conv5` 또는 `block6` 이후만 풀어서 fine-tune하는 방식으로 옮겼다. E1의 val_acc 0.4042는 사실상 얼굴 이외 영역에 낚이는 상태. E2에서 bbox crop + conv5 unfreeze + 가벼운 aug를 넣자 0.7733으로 뛰었고, 백본을 EfficientNet-B0로 교체한 E4에서 0.7958까지 올라왔다. TF/Keras 스택, `scripts/train.py`.

### 4.3 ViT-B/16 — 2-stage 전략

`timm.create_model('vit_base_patch16_224', pretrained=True, num_classes=4)`로 86M 파라미터 규모. 학습은 2-stage 전략으로 갈랐다. Stage A는 backbone을 얼리고 head만 5 epoch 학습, Stage B는 전체를 풀어 15 epoch fine-tune. optimizer AdamW (wd 1e-4) + cosine schedule (warmup 2ep) + grad_clip 1.0 + bf16 AMP 조합. PyTorch 스택, `scripts/train_vit.py`. val_acc 0.8458로 현 best.

### 4.4 SigLIP + yua-encoder wrapper

팀 자체 설계한 vision_encoder (`yua-encoder`)를 그대로 살린 트랙. 구조는 SigLIP ViT-Base (freeze) → last_hidden_state [B, 576, 768] → PixelShuffle r=4 → 2-layer projection MLP → mean pool → Linear(512→4). projection+head 합쳐 약 25M 파라미터가 학습 대상이고, backbone은 얼린 상태로 linear probe에 가깝게 돌린다. 입력 해상도 384. PyTorch + transformers 스택, `scripts/train_siglip.py`. **E6 학습 진행 중** (본 문서 작성 시점), 결과는 추후 5.1 표에 추가 예정.

### 4.5 공통 학습 설정

세 트랙이 공유하는 부분만 정리하면 다음과 같다. optimizer는 AdamW (weight_decay 1e-4), learning rate schedule은 CosineDecay with warmup (E3 이후 2 epoch warmup 고정), gradient clipping 1.0, label smoothing 0.1 (CNN 트랙 E2~E4), mixed precision bf16 (ViT·SigLIP). 입력 정규화는 ImageNet mean/std. 재현성은 `tf.keras.utils.set_random_seed` / `torch.manual_seed` + `--name` prefix로 잡았다.

| 트랙 | 모델 | 파라미터 | 학습가능 | 입력 | 주요 hyperparameter |
|---|---|---:|---:|---:|---|
| A | ResNet50 (frozen) | 23.5M | ~8K | 224 | lr 1e-3, bs 128 |

### 5.1 E1~E6 결과 비교

| # | 모델 | val_acc | val_loss | gap (train−val) | 핵심 기법 |
|---|---|---:|---:|---:|---|
| E1 | ResNet50 frozen | 0.4042 | 1.3132 | 0.08 | FC만 학습, bs 128, aug 없음 |
| E2 | ResNet50 ft (conv5) | 0.7733 | 0.8422 | 0.21 | + face crop + 중간 aug + bs 32 + ls 0.1 |
| E3 | ResNet50 ft (conv5) + strong reg | 0.7617 | 0.9941 | 0.15 | AdamW + cosine_warmup + dropout 0.5 + ls 0.2, monitor=val_loss |
| E4 | **EfficientNet-B0 ft (block6)** | **0.7958** | 0.7526 | 0.15 | AdamW + cosine_warmup 2, 중간 aug |
| E5 | **ViT-B/16 2-stage** | **0.8458** | 0.5234 | — | Stage A(freeze 5ep) → Stage B(unfreeze 15ep, AdamW, cosine_warmup 2, grad_clip 1.0, bf16) |

### 5.2 학습 곡선 — Stage 전환점의 의미

E5에서 가장 선명하게 드러난 패턴이 Stage A → Stage B 전환 직후의 점프다. Stage A 5 epoch 동안 val_acc는 0.73대에서 느리게 수렴하다가, Stage B 첫 1~2 epoch에 0.81 근처까지 뛰어올랐다. freeze 상태의 head가 대략의 feature 공간을 잡아둔 뒤 backbone을 풀면, 전체 네트워크가 emotion-specific representation 쪽으로 빠르게 이동한다는 해석이 자연스럽다. ViT는 첫 epoch compile에 약 2.5분이 걸리지만 이후 epoch당 140초 내외로 안정화되어 전체 20 epoch이 GPU 1장으로 현실적인 시간 안에 돌았다.

CNN 트랙 쪽에서는 E2의 gap 0.21이 눈에 띈다. train이 빠르게 치고 올라가는 동안 val이 따라붙지 못한 전형적인 overfit 신호였고, E3에서 dropout 0.5 + label smoothing 0.2 + AdamW로 억눌러 gap을 0.15까지 내렸다. 다만 val_acc는 0.7617로 오히려 떨어졌다 — 뒤에서 다시 짚는다.

### 5.3 Ablation 의의

구간별로 어느 변수가 실제로 movable했는지 정리해 보면 다음과 같다.

**E1 → E2 (+37%p)**. face crop 도입이 결정적이었다. bbox crop이 빠진 상태에서는 고해상도 원본을 224로 다운샘플링하며 얼굴 픽셀이 수십 제곱 픽셀 수준까지 줄어드는 경우가 생긴다. crop으로 얼굴을 먼저 잘라낸 뒤 resize하니 같은 ResNet50에서도 성능이 두 배 가까이 올랐다. 동시에 conv5 unfreeze와 가벼운 aug도 들어갔기 때문에 세 변수의 합산이긴 한데, 각각 빼면서 돌려본 ablation 결과 crop의 단독 기여가 가장 컸다.

**E2 → E3 (−1.2%p)**. strong regularization은 gap을 축소하는 데 성공했으나 val_acc 절대값은 오히려 줄었다. 원인 진단: label_smoothing=0.2와 monitor=val_loss 조합이 문제였다. ls 0.2는 loss 하한을 올려 best model 선택 시점을 실제 val_acc 피크와 어긋나게 만들었고, monitor=val_loss 기준으로 체크포인트를 잡다 보니 val_acc 최고점을 놓쳤다. E4부터 monitor=val_accuracy로 되돌려 문제를 제거했다.

**E3 → E4 (+3.4%p)**. 백본 교체(ResNet50 → EfficientNet-B0)가 hyperparameter 조정보다 효과가 컸다. gap은 E3와 동일한 0.15를 유지하면서 val_acc 절대값을 0.7958까지 끌어올렸다 — overfit 억제와 성능 향상을 동시에 달성한 유일한 구간.

**E4 → E5 (+5%p)**. ViT-B/16의 전역 self-attention이 CNN 대비 명확한 우위를 보였다. 얼굴 표정은 눈·입·눈썹처럼 서로 떨어진 영역의 상관관계가 중요한데, CNN은 local receptive field를 여러 층 쌓아 간접적으로 포착하는 반면 ViT는 첫 블록부터 전역 토큰 interaction을 계산한다. 데이터 7천 장이면 ViT에 불리한 규모라는 통념과 달리, pretrained weight가 워낙 강해서 2-stage fine-tune만으로도 CNN 최고 기록을 넘겼다.

### 5.4 시행착오 (폐기한 설정들)

처음 E2를 굴릴 때 TF의 `RandomCrop` 을 파이프라인 뒤쪽에 넣었다가 face crop으로 잘라 놓은 얼굴이 한 번 더 잘리는 문제가 생겼다. 입 아래가 잘려나가는 케이스를 샘플 시각화에서 확인한 뒤 `RandomResizedCrop(scale=0.9~1.0)` 로 교체했다. E3 설계 시점에는 AdamW + strong reg 조합이 순수 성능까지 끌어올려 줄 것으로 기대했는데, 앞서 적은 대로 val_acc에서 오히려 0.0116 손해를 봤다 — monitor 지표 선택이 hyperparameter만큼 중요하다는 교훈. 인프라 쪽에서는 cuDNN 9.1 ↔ 9.3 호환성 이슈가 있었다. TF 2.21이 cuDNN 9.3+를 요구하는데 timm을 설치하면 의존성이 9.1로 내려가는 충돌. 실측해 보니 양쪽 버전 모두에서 TF가 정상 동작해 추가 작업은 없이 넘어갔다.

### 5.5 성능 상한 추정

annot_A/B/C 3명 라터의 라벨이 어디까지 일치하는지를 EDA에서 뽑았다. 3인 전원일치 비율 0.6108, 다수결(2/3 이상) 일치 비율 0.9045. 사람 라터조차 완벽히 동의하지 못하는 샘플이 40%에 가까운 셈이고, 모델의 val_acc 상한은 이 두 숫자 사이에 있다고 보는 게 타당하다. 즉 0.85~0.90 구간이 현실적 ceiling이고, E5의 0.8458은 이미 그 하단에 근접한 위치다. 앙상블 E8로 0.86~0.87, 잘 풀리면 0.88까지 노려볼 수 있지만 그 이상은 라벨 노이즈에 막힐 가능성이 크다. soft-label (3인 vote 분포를 직접 학습 target으로 쓰는) 실험이 의미 있을 것으로 보이며 5/1 이후 추가 실험 후보로 둔다.

## 6. 서비스 프로토타입 (5/1~5/7)

### 6.1 시스템 구조

런타임 흐름을 블록 다이어그램으로 정리하면 다음과 같다.

```
[사용자]
    │  상황 요청
    ▼
[챗봇]  (초기엔 정적 스토리 세트, 이후 LLM API)
    │  감정이 자연스럽게 드러나는 상황 카드 제시
    │  (내부에는 expected_emotion 라벨이 붙어 있음)
    ▼
[사용자 표정 촬영]  (웹캠 or 모바일)
    │
    │  face crop (annot_A와 동일한 전처리) + EXIF 정규화 + 224/384 resize
    ▼
[감정 분류 모델]  (E5 best 단일 or E8 앙상블)
    │
    │  (pred, confidence) 출력
    ▼
[분기 로직]
    │
    ├── pred == expected AND conf ≥ 0.85 ──▶ 스토리 다음 장면
    ├── conf < 0.6 ──▶ "다시 해볼까?" 재시도 루프
    └── 나머지 ──▶ 힌트 장면 (expected 감정의 예시 표정 노출)
         │
         │  (pred == expected AND conf ≥ 0.85 인 샘플만)
         ▼
    [rehearsal buffer]  →  nightly 부모/치료사 검수  →  승인 샘플 재학습
```

### 6.2 MVP 구현 범위 — 무엇까지 만들고 어디서 멈추는가

- **감정 분류 모듈**: 완성. E5 단일 (val_acc 0.8458) 또는 E8 앙상블 (목표 0.86+)을 `predict(img_path) → (label, confidence)` 인터페이스로 노출. 실서비스 레이턴시 목표 500ms 이내 (ViT-B/16 단일이면 A40에서 50~80ms, 프로토는 CPU 추론도 감안).
- **챗봇 모듈**: 시뮬레이션 수준. 1차 MVP는 감정별 3~5개씩 수작업으로 작성한 정적 스토리 세트 (총 16~20개 카드)를 JSON으로 관리한다. 5/6 이후 여유가 있으면 LLM API를 연동해 동적 스토리 생성까지 붙일 계획.
- **분기 로직**: 규칙 기반. 위 다이어그램의 3분기 if-else 그대로. 상태는 Streamlit session_state에 스택으로 쌓는다.
- **프론트엔드**: Streamlit. 좌측 스토리 진행 로그, 우측 웹캠 프리뷰 + 캡처 버튼 + 결과 (softmax 막대).

### 6.3 응용 시나리오

교육 영역에서는 LMS 통합을 가정한다. 스토리 하나가 한 학습 단위가 되고, 정답 감정을 맞출 때까지 반복하는 연습 모듈로 쓰인다. 부모·치료사 대시보드에 아이의 감정별 정확도·재시도 횟수·세션 길이가 누적되는 형태. 게임 영역에서는 표정이 곧 컨트롤러가 되는 스토리 분기형 게임이 가능하다. 분기점마다 클릭이 아닌 표정이 입력되는 것만 다르지 구조 자체는 비주얼 노벨 장르와 동일하다.

### 6.4 자가발전 파이프라인 — 단계 1 한정

단계 1은 "조건부 수집까지"에 그친다. 세션 중에 아래 두 조건을 모두 만족한 샘플만 `rehearsal_buffer/`에 얼굴 crop과 expected 라벨을 쌍으로 저장한다.

```
if conf ≥ 0.85 and pred == expected_emotion:
    buffer.add(crop, expected_emotion, session_meta)
```

nightly 배치에서 부모 또는 치료사가 웹 UI로 훑으며 approve/reject를 클릭. 승인된 샘플만 원본 학습셋의 일부와 섞어 LwF (Learning without Forgetting) 손실로 소량 재학습한다. 여기서 self-poisoning을 막는 장치가 3중으로 박혀 있다 — (a) confidence threshold 0.85, (b) pred == expected 일치, (c) 사람 검수 게이트. 어느 하나라도 통과 못 하면 버퍼에 쌓이지 않거나 학습에 투입되지 않는다.

실시간 재학습 (세션 도중 모델 업데이트)은 의식적으로 피했다. 레이턴시 리스크와 사용자별 drift를 동시에 감당할 여력이 MVP에는 없다. 단계 2 (개인화 fine-tune, 사용자별 LoRA)는 향후 과제 슬라이드에 로드맵으로만 올린다.

## 7. 한계와 윤리

### 7.1 모델 한계

4-class로 잘라 학습했기 때문에 혼합감정 (기쁨과 놀람이 섞인 경우 등), 감정의 **강도** (약한 화남 vs 격분), 그리고 중립·경멸·혐오 같은 라벨은 애초에 다룰 수 없다. 3rater 일치율 (3.7절)을 근거로 추정한 성능 천장이 0.85~0.90 구간이라는 점도 그대로 한계다. E5 단일이 이미 0.8458으로 천장 하단에 붙어 있어서, 앙상블·seg·soft label을 모두 투입해도 실질적 개선폭은 1~3%p 수준이 현실적이다. 더해서 4/30 제출 이후 외부에서 별도 제공될 test set에 대한 성능은 현 시점에 알 수 없고, val ↔ test 분포 차이에서 drop이 발생할 가능성을 배제할 수 없다.

### 7.2 데이터 한계

이스트소프트 데이터셋은 한국인 위주로 구성되어 있어 타 인종·문화권 얼굴에 대한 일반화를 보장하지 못한다. 연령 분포도 mean 26.4, median 20으로 성인 쪽에 쏠려 있는데, 1차 타깃 사용자가 아동이라는 점과 정면으로 충돌한다. 학습 시점엔 성인 얼굴로 학습한 모델이 아동 얼굴에서 체계적 편향 (예: 아동의 큰 눈·작은 얼굴에서 panic을 과다 예측)을 보일 위험이 있다. 이 갭을 어떻게 메울지는 8장(향후 과제)에서 다시 짚는다.

### 7.3 개인정보·법적 리스크

서비스의 1차 사용자가 **미성년 아동**이라는 점에서 개인정보 처리는 일반 서비스보다 한 단계 엄격한 기준을 적용해야 한다. 2025년 6월 발효된 미국 FTC COPPA 개정안은 아동 얼굴 이미지 같은 biometric 식별자를 수집·저장·학습에 사용할 때 **verifiable parental consent** (검증 가능한 보호자 동의)를 의무화했다. 국내에서도 개인정보보호법 상 만 14세 미만 아동 정보에는 법정대리인 동의가 필요하다. 프로토타입 단계에서도 (a) 가명처리, (b) 얼굴 원본이 아닌 crop·feature만 buffer에 저장, (c) 사용자별 삭제권 UI, (d) 온디바이스 추론 옵션 — 이 네 가지를 기본 요건으로 명시한다.

### 7.4 자가발전 특유의 위험

6.4절의 자가발전 파이프라인은 self-poisoning — 모델이 잘못 예측한 샘플을 정답이라고 믿고 다시 학습해 편향을 심화시키는 — 리스크를 본질적으로 안고 있다. 방어는 앞서 적은 3중 게이트 (confidence 0.85 + expected 일치 + 사람 검수)에 의존하는데, threshold가 너무 보수적이면 데이터가 거의 안 쌓이고, 너무 관대하면 노이즈가 섞인다. 추가로 "억지 표정"의 위험도 있다. 아동이 스토리를 진행시키고 싶어서 실제로 느낀 감정과 무관하게 기쁨 표정을 연기할 경우, 이 샘플은 (conf ≥ 0.85, pred == expected)를 통과해 buffer에 들어간다. 이걸 어떻게 거를지는 현재 미해결이며, 중간 발표 이후 연구 과제로 다룬다.

### 7.5 윤리적 서술 톤

서비스 설명에서 "자폐 아동을 고친다"는 식의 의료적 언어는 절대 쓰지 않는다. 목표는 **학습 보조**이지 치료가 아니며, 현장 도입 시 반드시 임상 전문가의 감독 아래 쓰이는 것을 전제로 한다. 이 포지셔닝을 기획안·발표자료·README 전반에서 일관되게 유지한다.

## 8. 회고와 향후 과제

### 8.1 잘 풀린 지점

**face crop + conv5 unfreeze 조합**이 예상보다 훨씬 컸다. E1 → E2 구간에서 +37%p (0.4042 → 0.7733). 고해상도 원본 (중앙값 3088×2208)을 224로 그냥 리사이즈하면 얼굴 픽셀이 뭉개진다는 가설을 세우고 들어간 결과였는데, 숫자로 이렇게까지 뛸 줄은 몰랐다. 기획안 단계에서 "crop이 점수랑 직결될 것" 문단을 강하게 적어 둔 게 E1 실패 후 빠르게 방향 전환하는 데 도움이 됐다.

**ViT-B/16의 전역 attention 우위**. CNN 최고 기록 (E4, EfficientNet-B0, 0.7958)을 ViT가 2-stage fine-tune 한 번 돌려 0.8458로 넘겼다. 7천 장 규모면 ViT에 불리하다는 통념과 달리, pretrained weight의 힘과 2-stage 전략 (Stage A freeze 5ep → Stage B unfreeze 15ep)이 맞물리면서 차이를 벌렸다. 눈·입·눈썹처럼 떨어진 영역의 상관관계를 보는 문제에서 self-attention이 local conv보다 유리하다는 가설이 깔끔하게 관측됐다.

**자체 SigLIP wrapper도 동작했다**. yua-encoder (vision_encoder + PixelShuffle + projection MLP) 구조가 linear probe로 0.8192를 찍어 E4 EfficientNet을 상회. 팀이 직접 설계한 컴포넌트가 downstream에서 제대로 붙는 걸 숫자로 증명했다는 점이 창의성 항목에서 중요했다.

### 8.2 시행착오와 폐기한 설정

한 세트로 돌아간 것만 쓴 게 아니다. 폐기한 설정 몇 개를 기록으로 남긴다.

monitor=val_loss + label_smoothing 0.2 조합 (E3)에서 best checkpoint가 실제 val_accuracy 피크와 어긋났다. ls가 loss 하한을 올려 놓은 상태에서 val_loss 기준으로 체크포인트를 고르니, val_acc 최고점 epoch이 아니라 loss가 가장 낮은 (accuracy는 살짝 떨어진) epoch이 저장됐다. E4부터 monitor=val_accuracy로 되돌려 해결. hyperparameter보다 monitor 지표 선택이 먼저라는 교훈.

RandomCrop이 bbox crop한 얼굴에 한 번 더 걸리는 버그. E2 초기에 샘플 시각화에서 입 아래가 잘려나간 크롭을 발견하고 `RandomResizedCrop(scale=0.9~1.0)`로 교체. 시각화 안 했으면 몇 epoch 더 돌릴 뻔했다.

ViT 첫 epoch의 compile 오버헤드가 20분 넘게 잡혀서 당황. 2번째 epoch부터 140초 내외로 안정화되길래 환경 문제는 아니고 torch compile 캐시 이슈로 확인. 대규모 sweep 돌릴 땐 warm-up run 하나 미리 돌려두는 패턴으로 회피.

TF 2.21이 cuDNN 9.3+를 요구하는데 timm 설치하면 9.1로 내려가는 버전 충돌. 실측해 보니 양쪽 다 돌길래 그냥 넘어갔지만, requirements.txt를 지금 상태로 배포하면 다른 환경에서 재현 안 될 가능성은 남아 있다.

### 8.3 향후 과제

**Soft label 학습** — 3rater 라벨 분포를 target으로 쓰는 KL divergence 방식. annot_A 하나만 쓰는 현 방식이 val_acc 0.85~0.90 천장에 막혀 있다면, 이론적 상한 자체를 올릴 방법은 이쪽이다. 5/1 이후 E9 실험 후보로 등록.

**아동 얼굴 데이터 보강** — 학습셋 연령 median 20이 1차 타깃 사용자 (아동)와 어긋나는 문제 해결책. 공개 아동 얼굴 데이터셋 (예: CAFE, NIMH-ChEFS) 추가 학습 or few-shot fine-tune.

**챗봇 LLM 연동** — 현 MVP의 정적 스토리 세트 (16~20개)를 넘어 동적 스토리 생성. LLM API로 expected_emotion을 조건부로 주고 스토리 생성하는 프롬프트 설계.

**자가발전 단계 2 — HITL 파이프라인 실구현**. 부모·치료사용 검수 UI (Streamlit 확장 or 별도 웹앱), LwF 손실 기반 nightly 재학습 스크립트, 재학습 전후 val_acc 비교 후 자동 rollback 로직. 윤리 게이트 (동의 UI, 삭제권) 포함.

**Test set 공개 후 POC** — 4/30 이후 외부에서 공개될 test set에서의 성능을 받고, 실제 서비스 환경 (웹캠 프레임, 조명 변동, 각도 다양성)을 가정한 stress test를 설계. 기획안에 적은 "데모에서 쓸 만한 정확도" 연구 질문의 마지막 검증이 여기에 들어간다.
